# Page Plan Patterns

Use this reference when deciding whether a page needs a full script, a shared group script, a quick-flip cue, or a user-edit warning.

## Page types

| Page type | Script strategy | `next_step` strategy |
| --- | --- | --- |
| Title, cover, or chapter divider | If it is page 1, use a soft-start sentence. For later dividers, use one audience-facing framing sentence. Do not say "from this page". | Pause, then move to the first content page. |
| Concept page | Translate the concept into one usable judgment. | Move to the page that proves it. |
| Master plan | If the plan alone causes meta-talk, combine it with the next circulation, zoning, or node page. | Tell the presenter which next page completes the explanation. |
| Diagram or analysis page | Explain the decision logic, not the drawing operation. | Move after the logic is clear. |
| Technical control page | State the constraint and design response. Avoid pretending to know unshown approvals. | Mark needs-user-edit when exact cause matters. |
| Scheme implementation page | Name tangible objects, materials, nodes, and visible design actions before describing atmosphere. | Mark needs-user-edit if exact material or process must be checked. |
| Effect image group | First page gets the shared speech with real objects/materials. Later pages name only new visible differences. | Give a page range and flip rhythm. |
| Homogeneous implementation pages | Use fewer fixed words, but keep the words concrete. Leave room for live design judgment. | Use quick-flip, free-explanation, or supporting-view cues. |
| Uncertain factual page | Do not invent. Explain only what is visible. | Mark needs-user-edit and name what should be checked. |

## Status labels

Localize labels into the user's language. Keep them short because the sidebar is narrow.

Common labels:

- opening hold
- chapter turn
- priority review
- needs-user-edit
- continuous flip
- supporting view
- free explanation
- quick flip
- phase close

## First page soft-start

Treat page 1 as a rhythm-setting page, not a proof page. The first script should let a real presenter open calmly, name the project, define the reporting scope, and lightly plant the main thread.

Choose the opening pattern by file type:

- `稳妥软启动`: formal, government, client, or review context.
- `情绪钩子软启动`: spatial, brand, culture-tourism, or public-facing context where a restrained hook helps.
- `高度总结软启动`: long, complex, leadership, or comparison context where the audience needs a quick orientation.

Do not overload page 1 with materials, nodes, parameters, or full strategy proof unless the first source page already contains those technical facts.

## Repeated groups

When several pages show the same space:

1. Write one shared audience-facing sentence on the first page.
2. On following pages, write one short support sentence only if the viewpoint adds visible information: material, furniture, planting, lighting, wall, edge, node, structure, process, or operation.
3. Put page movement in `next_step`.
4. Do not invent a new design claim for every similar view.
5. Do not repeat only concept words. Rendering pages should explain what physically makes the effect happen.

Use this when the next pages differ only by:

- angle
- distance
- day or night mood
- nearby planting
- people in the rendering
- minor material visibility

## Combining pages

Combine adjacent pages when a single page would force presenter coaching.

Examples:

- master plan plus circulation analysis
- concept phrase plus space structure
- location map plus first effect image
- renovation comparison plus exact project note

The first page should give the audience-facing anchor. The cue should explain that the next page completes the point.

## User-edit triggers

Mark a page for user edit when the script requires facts that are not safely visible in the PDF:

- why an area was changed
- what the client approved
- which standard or policy applies
- exact redline or boundary meaning
- exact node priority
- final material or construction choice

The script may still describe the visible relation. The cue should name what the user must check.
